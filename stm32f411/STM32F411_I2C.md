## 1. Introduction to I2C

**I2C** (Inter-Integrated Circuit, pronounced "I squared C" or "I two C") is a synchronous, multi-master, multi-slave serial communication protocol invented by Philips (now NXP) in 1982. It uses just **two wires** — SDA (data) and SCL (clock) — to connect multiple devices on the same bus, each identified by a unique address.

**Key characteristics:**
- Two-wire bus (SDA + SCL) plus common ground
- Synchronous — the master generates the clock
- Half-duplex — master and slave share the SDA line (not simultaneous TX and RX)
- Multi-slave — up to 128 devices (7-bit addressing) or 1024 devices (10-bit addressing)
- Multi-master capable — multiple masters can coexist with arbitration
- Open-drain bus — requires pull-up resistors
- Speeds: Standard (100 kHz), Fast (400 kHz), Fast-Plus (1 MHz), High-Speed (3.4 MHz), Ultra-Fast (5 MHz)
- Built-in acknowledgment for error detection

**Comparison with SPI and UART:**

| Feature        | I2C          | SPI           | UART          |
|----------------|--------------|---------------|---------------|
| Wires          | 2            | 4+            | 2             |
| Speed          | Up to 400kHz (std) | Up to 100 MHz | Up to 5+ Mbps |
| Devices        | 128+         | One per CS    | 2 only        |
| Addressing     | Built-in     | Via CS pin    | None          |
| Complexity     | Medium       | Low           | Low           |
| Duplex         | Half         | Full          | Full          |

## 2. STM32F411 I2C Hardware Overview

The STM32F411 has **3 I2C peripherals**:

| Peripheral | APB Bus | Max Speed     | Default Pins                |
|------------|---------|---------------|-----------------------------|
| I2C1       | APB1    | 400 kHz (FM)  | PB6 (SCL), PB7 (SDA)        |
| I2C2       | APB1    | 400 kHz (FM)  | PB10 (SCL), PB11 (SDA)      |
| I2C3       | APB1    | 400 kHz (FM)  | PA8 (SCL), PC9 (SDA)        |

All I2C peripherals are on APB1, which runs at up to 42 MHz on the STM32F411.

**Key hardware features:**
- 7-bit and 10-bit addressing support
- DMA requests for both TX and RX
- SMBus (System Management Bus) 2.0 support
- Packet error checking (PEC) for SMBus
- Clock stretching (slave can hold SCL low to slow the master)
- Broadcast call (address 0x00) support
- Wakeup from STOP mode (WUFSI feature)

## 3. I2C Bus Signals and Electrical Characteristics

### Open-Drain Bus

I2C uses open-drain (open-collector) signaling. Devices can only pull the line LOW; external pull-up resistors pull the lines HIGH when no device is pulling them down.

This allows:
- Wire-AND logic: any device can pull the line low (used for arbitration and ACK)
- Safe multi-driver bus — no short-circuit if two devices drive simultaneously

### Pull-Up Resistor Calculation

```
R_pull-up = (VCC - V_OL) / I_sink
```

Where:
- `VCC` = supply voltage (3.3 V on STM32F411)
- `V_OL` = maximum output low voltage (~0.4 V)
- `I_sink` = current capability of the weakest driver (~3 mA typical)

**Practical formula considering bus capacitance:**

```
R_min = (VCC - V_OL_max) / I_OL  [ensure LOW level]
R_max = t_rise / (0.8473 × C_bus)  [ensure rise time spec is met]
```

**Typical values:**
- Standard Mode (100 kHz), short bus (<1m): 4.7 kΩ to 10 kΩ
- Fast Mode (400 kHz), moderate bus: 2.2 kΩ to 4.7 kΩ
- Fast-Plus Mode (1 MHz): 1 kΩ
- Many modules (OLED, IMU boards) include on-board pull-ups (typically 4.7 kΩ)

### GPIO Configuration

Both SDA and SCL pins must be configured as **open-drain, alternate function** with NO internal pull-up (pull-ups are external):

```c
// GPIO for I2C: open-drain, alternate function, high speed, no pull
GPIO->OTYPER  |=  (1 << pin);        // Open-drain
GPIO->OSPEEDR |=  (3 << (pin*2));    // High speed
GPIO->PUPDR   &= ~(3 << (pin*2));    // No pull (external pull-up)
GPIO->MODER   &= ~(3 << (pin*2));
GPIO->MODER   |=  (2 << (pin*2));    // Alternate function
GPIO->AFR[n]  |=  (AF4 << (bit));    // AF4 for I2C1/2/3
```

## 4. I2C Protocol — Frame Format

### Basic Write Transaction (Master → Slave)

```
START | ADDR(7) | W(0) | ACK | DATA[0] | ACK | DATA[1] | ACK | ... | STOP
  S      A6-A0    0     AK    D7-D0    AK    D7-D0    AK         P
```

### Basic Read Transaction (Master ← Slave)

```
START | ADDR(7) | R(1) | ACK | DATA[0] | ACK | DATA[n-1] | NACK | STOP
  S      A6-A0    1     AK    D7-D0    AK    D7-D0      NAK      P
```

### Combined Write-then-Read (Register Read Pattern)

This is the most common I2C operation — write a register address, then read data:

```
START | ADDR | W | ACK | REG_ADDR | ACK | RESTART | ADDR | R | ACK | DATA | NACK | STOP
```

### I2C Conditions

| Condition  | Description                                             |
|------------|---------------------------------------------------------|
| START (S)  | SDA falls while SCL is HIGH                             |
| STOP (P)   | SDA rises while SCL is HIGH                             |
| RESTART    | Another START without a preceding STOP (repeated start) |
| ACK        | Receiver pulls SDA LOW during the 9th clock pulse       |
| NACK       | Receiver leaves SDA HIGH (or it's the last byte)        |
| Data bit   | Sampled on rising edge of SCL; must be stable while SCL=HIGH |

### 10-Bit Addressing

For 10-bit addresses (0x00–0x3FF), the master sends two address bytes:

```
START | 11110XX0 | ACK | XXXXXXXX | ACK | DATA...
       ^^^^^^^^ first byte: 5-bit magic + top 2 bits of address + W
                             ^^^^^^^^ second byte: lower 8 bits of address
```

## 5. I2C Addressing

### 7-Bit Address

The 7-bit address is sent as bits [7:1] of the first byte. Bit 0 is the direction: 0=write, 1=read.

So when you write `HAL_I2C_Master_Transmit(&hi2c1, 0x68 << 1, ...)`, you are left-shifting the 7-bit address by 1 to make room for the R/W bit.

**Common device addresses:**

| Device          | 7-bit Address | Notes                        |
|-----------------|--------------|------------------------------|
| MPU6050 IMU     | 0x68 or 0x69 | AD0 pin selects              |
| BME280          | 0x76 or 0x77 | SDO pin selects              |
| SSD1306 OLED    | 0x3C or 0x3D | SA0 pin selects              |
| AT24C EEPROM    | 0x50–0x57    | A0-A2 pins select            |
| PCF8574 I/O exp.| 0x20–0x27    | A0-A2 pins select            |
| DS3231 RTC      | 0x68         | Fixed                        |
| LIS3DH accel    | 0x18 or 0x19 | SDO/SA0 selects              |

### Reserved Addresses

The I2C specification reserves certain addresses:

| Address  | Purpose                              |
|----------|--------------------------------------|
| 0x00     | General call (broadcast)             |
| 0x01     | CBUS address                         |
| 0x78–0x7B| 10-bit address prefix                |
| 0x7C–0x7F| Reserved                             |

## 6. I2C Registers (Bare-Metal)

### I2C_CR1 — Control Register 1

| Bit | Name    | Description                                |
|-----|---------|--------------------------------------------|
| 15  | SWRST   | Software reset (clears peripheral)         |
| 11  | ACK     | Enable ACK generation after received byte  |
| 10  | STOP    | Generate STOP condition                    |
| 9   | START   | Generate START condition                   |
| 8   | NOSTRETCH| Disable clock stretching (slave only)     |
| 7   | ENGC    | Enable general call (address 0x00)         |
| 6   | ENPEC   | Enable packet error checking (SMBus)       |
| 5   | ENARP   | Enable ARP (SMBus)                         |
| 1   | SMBUS   | SMBus mode (1) vs I2C mode (0)             |
| 0   | PE      | Peripheral enable                          |

### I2C_CR2 — Control Register 2

| Bits | Name    | Description                                         |
|------|---------|-----------------------------------------------------|
| 12   | LAST    | DMA last transfer (generate NACK after last byte)   |
| 11   | DMAEN   | DMA request enable                                  |
| 10   | ITBUFEN | Buffer interrupt enable (TXE, RXNE events)          |
| 9    | ITEVTEN | Event interrupt enable (START, STOP, ADDR, BTF etc) |
| 8    | ITERREN | Error interrupt enable                              |
| 5:0  | FREQ    | APB1 frequency in MHz (must match actual frequency) |

**Important:** The FREQ field must be set to the actual APB1 frequency in MHz. For STM32F411 at max speed, FREQ = 42.

### I2C_SR1 — Status Register 1

| Bit | Name   | Description                                          |
|-----|--------|------------------------------------------------------|
| 15  | SMBALERT| SMBus alert                                         |
| 14  | TIMEOUT| Timeout or Tlow error                               |
| 12  | PECERR | PEC error in reception                              |
| 11  | OVR    | Overrun/underrun                                    |
| 10  | AF     | Acknowledge failure (NACK received)                 |
| 9   | ARLO   | Arbitration lost                                    |
| 8   | BERR   | Bus error (misplaced START/STOP)                    |
| 7   | TXE    | Data register empty (for transmit)                  |
| 6   | RXNE   | Data register not empty (data received)             |
| 4   | STOPF  | Stop detected (slave mode)                          |
| 3   | ADD10  | 10-bit address header sent                          |
| 2   | BTF    | Byte transfer finished                              |
| 1   | ADDR   | Address sent (master) / Address matched (slave)     |
| 0   | SB     | Start bit generated                                 |

### I2C_SR2 — Status Register 2

| Bit | Name    | Description                              |
|-----|---------|------------------------------------------|
| 7:1 | PEC     | Packet error checking register           |
| 4   | DUALF   | Dual flag (slave mode, dual address)     |
| 3   | SMBHOST | SMBus host header                        |
| 2   | SMBDEFAULT| SMBus device default address           |
| 1   | TRA     | Transmitter (1) / Receiver (0)           |
| 0   | BUSY    | Bus busy                                 |

**Critical:** Reading SR2 after SR1 clears the ADDR bit (part of the required sequence).

### I2C_CCR — Clock Control Register

| Bits  | Name | Description                                    |
|-------|------|------------------------------------------------|
| 15    | FS   | Mode: 0=Standard (100kHz), 1=Fast (400kHz)    |
| 14    | DUTY | Fast mode duty cycle: 0=2:1, 1=16:9           |
| 11:0  | CCR  | Clock control register value                   |

**Standard mode (FS=0):**
```
CCR = fAPB1 / (2 × fSCL)
For 100 kHz at 42 MHz APB1: CCR = 42,000,000 / (2 × 100,000) = 210
```

**Fast mode (FS=1, DUTY=0, 2:1 ratio):**
```
CCR = fAPB1 / (3 × fSCL)
For 400 kHz at 42 MHz APB1: CCR = 42,000,000 / (3 × 400,000) = 35
```

**Fast mode (FS=1, DUTY=1, 16:9 ratio):**
```
CCR = fAPB1 / (25 × fSCL)
For 400 kHz at 42 MHz APB1: CCR = 42,000,000 / (25 × 400,000) = 4 (minimum allowed = 1)
```

### I2C_TRISE — Rise Time Register

Specifies the maximum rise time of SDA/SCL in units of 1/fAPB1:

```
Standard mode: TRISE = (t_rise_max_ns / t_APB1_ns) + 1 = (1000 / (1000/42)) + 1 = 43
Fast mode:     TRISE = (300 / (1000/42)) + 1 = 13
```

## 7. Bare-Metal I2C Configuration

```c
/*
 * Configure I2C1 in Fast Mode (400 kHz)
 * APB1 = 42 MHz
 * SCL: PB6 (AF4), SDA: PB7 (AF4)
 */

void I2C1_Init(void)
{
    /* 1. Enable clocks */
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOBEN;   // GPIOB clock
    RCC->APB1ENR |= RCC_APB1ENR_I2C1EN;    // I2C1 clock

    /* 2. Configure PB6 (SCL) and PB7 (SDA) as AF4, open-drain, high speed */
    // Mode: Alternate function (10b)
    GPIOB->MODER &= ~((3U << (6*2)) | (3U << (7*2)));
    GPIOB->MODER |=  ((2U << (6*2)) | (2U << (7*2)));
    
    // Output type: Open-drain
    GPIOB->OTYPER |= (1U << 6) | (1U << 7);
    
    // Speed: High
    GPIOB->OSPEEDR |= ((3U << (6*2)) | (3U << (7*2)));
    
    // Pull: None (external pull-up resistors)
    GPIOB->PUPDR &= ~((3U << (6*2)) | (3U << (7*2)));
    
    // Alternate function: AF4 (I2C1)
    GPIOB->AFR[0] &= ~((0xFU << (6*4)) | (0xFU << (7*4)));
    GPIOB->AFR[0] |=  ((4U  << (6*4)) | (4U  << (7*4)));

    /* 3. Reset I2C peripheral */
    I2C1->CR1 |=  I2C_CR1_SWRST;
    I2C1->CR1 &= ~I2C_CR1_SWRST;

    /* 4. Set peripheral clock frequency (FREQ = APB1 in MHz = 42) */
    I2C1->CR2 = 42;  // Bits [5:0]

    /* 5. Set rise time */
    // Fast mode: TRISE = (300ns / (1/42MHz)) + 1 = 13
    I2C1->TRISE = 13;

    /* 6. Set CCR for 400 kHz fast mode, 2:1 duty */
    // CCR = 42000000 / (3 * 400000) = 35
    I2C1->CCR = I2C_CCR_FS | 35;  // FS=1, DUTY=0, CCR=35

    /* 7. Enable ACK and peripheral */
    I2C1->CR1 = I2C_CR1_PE | I2C_CR1_ACK;
}
```

## 8. Bare-Metal I2C Read/Write Operations

### Generate START and Send Slave Address

```c
static I2C_Status I2C1_Start(uint8_t addr_with_rw)
{
    uint32_t timeout = 10000;
    
    /* Wait until bus is free */
    while ((I2C1->SR2 & I2C_SR2_BUSY) && --timeout);
    if (!timeout) return I2C_TIMEOUT;
    
    /* Generate START */
    I2C1->CR1 |= I2C_CR1_START;
    
    /* Wait for SB (start bit generated) */
    timeout = 10000;
    while (!(I2C1->SR1 & I2C_SR1_SB) && --timeout);
    if (!timeout) return I2C_TIMEOUT;
    
    /* Send slave address + R/W bit (reading SR1 then writing DR clears SB) */
    I2C1->DR = addr_with_rw;
    
    /* Wait for ADDR (address sent, ACK received) */
    timeout = 10000;
    while (!(I2C1->SR1 & I2C_SR1_ADDR) && --timeout)
    {
        if (I2C1->SR1 & I2C_SR1_AF)  // NACK received
        {
            I2C1->CR1 |= I2C_CR1_STOP;
            I2C1->SR1 &= ~I2C_SR1_AF;
            return I2C_NACK;
        }
    }
    if (!timeout) return I2C_TIMEOUT;
    
    /* Clear ADDR by reading SR1 then SR2 */
    volatile uint32_t tmp = I2C1->SR1;
    tmp = I2C1->SR2;
    (void)tmp;
    
    return I2C_OK;
}

/* Write bytes to slave */
I2C_Status I2C1_WriteBytes(uint8_t devAddr, uint8_t regAddr, 
                            uint8_t *data, uint8_t len)
{
    I2C_Status status;
    
    /* START + send slave address with WRITE bit */
    status = I2C1_Start(devAddr << 1 | 0);  // 0 = write
    if (status != I2C_OK) return status;
    
    /* Send register address */
    I2C1->DR = regAddr;
    while (!(I2C1->SR1 & I2C_SR1_TXE));
    
    /* Send data bytes */
    for (uint8_t i = 0; i < len; i++)
    {
        I2C1->DR = data[i];
        /* Wait for TXE or BTF */
        uint32_t timeout = 10000;
        while (!(I2C1->SR1 & (I2C_SR1_TXE | I2C_SR1_BTF)) && --timeout);
        if (!timeout) return I2C_TIMEOUT;
        
        if (I2C1->SR1 & I2C_SR1_AF)  // NACK
        {
            I2C1->CR1 |= I2C_CR1_STOP;
            return I2C_NACK;
        }
    }
    
    /* Wait for BTF (last byte shifted out) */
    while (!(I2C1->SR1 & I2C_SR1_BTF));
    
    /* Generate STOP */
    I2C1->CR1 |= I2C_CR1_STOP;
    
    return I2C_OK;
}

/* Read bytes from slave (write reg address, restart, read) */
I2C_Status I2C1_ReadBytes(uint8_t devAddr, uint8_t regAddr,
                           uint8_t *data, uint8_t len)
{
    /* START + WRITE to send register address */
    I2C_Status status = I2C1_Start(devAddr << 1 | 0);
    if (status != I2C_OK) return status;
    
    I2C1->DR = regAddr;
    while (!(I2C1->SR1 & I2C_SR1_BTF));
    
    /* RESTART + READ */
    I2C1->CR1 |= I2C_CR1_START;  // Repeated START
    while (!(I2C1->SR1 & I2C_SR1_SB));
    
    I2C1->DR = (devAddr << 1) | 1;  // 1 = read
    while (!(I2C1->SR1 & I2C_SR1_ADDR));
    
    if (len == 1)
    {
        /* For 1 byte: disable ACK before clearing ADDR */
        I2C1->CR1 &= ~I2C_CR1_ACK;
        volatile uint32_t tmp = I2C1->SR1; tmp = I2C1->SR2; (void)tmp;
        /* Generate STOP */
        I2C1->CR1 |= I2C_CR1_STOP;
        /* Wait for data */
        while (!(I2C1->SR1 & I2C_SR1_RXNE));
        data[0] = I2C1->DR;
    }
    else if (len == 2)
    {
        /* For 2 bytes: set POS and disable ACK before clearing ADDR */
        I2C1->CR1 |= I2C_CR1_POS;
        I2C1->CR1 &= ~I2C_CR1_ACK;
        volatile uint32_t tmp = I2C1->SR1; tmp = I2C1->SR2; (void)tmp;
        /* Wait for BTF (both bytes received) */
        while (!(I2C1->SR1 & I2C_SR1_BTF));
        I2C1->CR1 |= I2C_CR1_STOP;
        data[0] = I2C1->DR;
        data[1] = I2C1->DR;
        I2C1->CR1 &= ~I2C_CR1_POS;
    }
    else
    {
        /* For N > 2 bytes */
        volatile uint32_t tmp = I2C1->SR1; tmp = I2C1->SR2; (void)tmp;
        
        for (uint8_t i = 0; i < len; i++)
        {
            if (i == len - 3)
            {
                /* Before second-to-last byte: wait for BTF, disable ACK */
                while (!(I2C1->SR1 & I2C_SR1_BTF));
                I2C1->CR1 &= ~I2C_CR1_ACK;
                data[i++] = I2C1->DR;
                /* Before last byte: wait for BTF, generate STOP */
                while (!(I2C1->SR1 & I2C_SR1_BTF));
                I2C1->CR1 |= I2C_CR1_STOP;
                data[i++] = I2C1->DR;
                data[i]   = I2C1->DR;
                break;
            }
            while (!(I2C1->SR1 & I2C_SR1_RXNE));
            data[i] = I2C1->DR;
        }
    }
    
    /* Re-enable ACK for next transaction */
    I2C1->CR1 |= I2C_CR1_ACK;
    
    return I2C_OK;
}
```

## 10. HAL I2C — Polling Mode

```c
/* ---- Write to a device register ---- */
uint8_t HAL_I2C_Write_Register(uint8_t devAddr, uint8_t regAddr, uint8_t value)
{
    uint8_t buffer[2] = { regAddr, value };
    HAL_StatusTypeDef status;
    
    // devAddr is 7-bit; HAL expects it left-shifted (<<1) or already shifted
    // CubeMX HAL: pass devAddr already shifted: (0x68 << 1)
    status = HAL_I2C_Master_Transmit(&hi2c1, devAddr << 1, buffer, 2, HAL_MAX_DELAY);
    return (status == HAL_OK) ? 0 : 1;
}

/* ---- Read from a device register (Memory Read API — simpler) ---- */
uint8_t HAL_I2C_Read_Register(uint8_t devAddr, uint8_t regAddr, uint8_t *data, uint16_t len)
{
    HAL_StatusTypeDef status;
    
    // HAL_I2C_Mem_Read handles the write-register-address + restart + read sequence
    status = HAL_I2C_Mem_Read(
        &hi2c1,
        devAddr << 1,           // Device address (shifted)
        regAddr,                // Register address to read from
        I2C_MEMADD_SIZE_8BIT,   // Register address size
        data,                   // Output buffer
        len,                    // Number of bytes to read
        HAL_MAX_DELAY           // Timeout
    );
    return (status == HAL_OK) ? 0 : 1;
}

/* ---- Memory Write API ---- */
uint8_t HAL_I2C_Write_Register_Multi(uint8_t devAddr, uint8_t regAddr, 
                                      uint8_t *data, uint16_t len)
{
    return (HAL_I2C_Mem_Write(
        &hi2c1,
        devAddr << 1,
        regAddr,
        I2C_MEMADD_SIZE_8BIT,
        data,
        len,
        HAL_MAX_DELAY
    ) == HAL_OK) ? 0 : 1;
}

/* ---- Check if device is present (ACK scan) ---- */
HAL_StatusTypeDef I2C_ScanDevice(uint8_t devAddr)
{
    return HAL_I2C_IsDeviceReady(&hi2c1, devAddr << 1, 3, 100);
}

/* I2C bus scanner — scan all 128 addresses */
void I2C_BusScan(void)
{
    printf("I2C Bus Scan:\r\n");
    for (uint8_t addr = 1; addr < 128; addr++)
    {
        if (HAL_I2C_IsDeviceReady(&hi2c1, addr << 1, 1, 10) == HAL_OK)
        {
            printf("  Device found at 0x%02X\r\n", addr);
```

## 14. I2C Clock Stretching

Clock stretching is a mechanism where a **slave device holds SCL LOW** to pause the master, giving itself more time to prepare data. The master must wait and cannot proceed until SCL is released.

**When clock stretching occurs:**
- After address recognition, before the slave is ready to send/receive
- Between bytes if the slave needs processing time
- After the last byte of a write before the slave can respond

**Disabling clock stretching:**
Some fast sensors don't support clock stretching. To disable it on the slave side:
```c
hi2c1.Init.NoStretchMode = I2C_NOSTRETCH_ENABLE;
```

**Detecting a clock-stretched bus (stuck SCL):**
If a device fails mid-transaction, SCL or SDA may be stuck. Recovery procedure:
```c
/* Bit-bang recovery: toggle SCL 9 times to release stuck slave */
void I2C_BusRecovery(void)
{
    // Configure SCL as GPIO output
    // Toggle SCL 9 times
    for (int i = 0; i < 9; i++)
    {
        HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_SET);
        HAL_Delay(1);
        HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_RESET);
        HAL_Delay(1);
    }
    // Generate STOP
    HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_SET);
    HAL_Delay(1);
    HAL_GPIO_WritePin(GPIOB, GPIO_PIN_7, GPIO_PIN_SET);
    // Reconfigure as I2C AF
    MX_I2C1_Init();
}
```

## 15. I2C Multi-Master and Arbitration

Multiple masters can exist on the same I2C bus. Arbitration happens automatically at the hardware level:

1. Both masters generate START simultaneously
2. Both transmit their first address byte, monitoring SDA while doing so
3. A master that transmits HIGH but reads LOW on SDA has lost arbitration — it stops immediately (the other master's LOW dominates)
4. The winning master continues uninterrupted; the losing master waits and retries

Arbitration is detected via the **ARLO (Arbitration Lost)** flag in SR1. In HAL, this triggers `HAL_I2C_ErrorCallback()` with `HAL_I2C_ERROR_ARLO`.

```c
void HAL_I2C_ErrorCallback(I2C_HandleTypeDef *hi2c)
{
    if (HAL_I2C_GetError(hi2c) & HAL_I2C_ERROR_ARLO)
    {
        // Arbitration lost — another master won
        // Wait a random time, then retry
        HAL_Delay(1 + (rand() % 5));
        RestartTransaction();
    }
}
```

## 16. Common Pitfalls and Debugging

### 1. Device not found (NACK on address)
- Verify pull-up resistors are present (2.2–10 kΩ between SDA/SCL and VCC)
- Confirm the device address — use an I2C scanner
- Check supply voltage to the device
- Verify GPIO alternate function (should be AF4 for I2C1/2/3)
- Confirm open-drain configuration on GPIO

### 2. Stuck SDA (SDA stuck LOW)
- A power-cycle reset may fix it temporarily
- Use the 9-clock recovery procedure
- Often caused by a master resetting mid-transaction

### 3. BUSY flag stuck HIGH after error
- Do a software reset: `I2C1->CR1 |= I2C_CR1_SWRST; I2C1->CR1 &= ~I2C_CR1_SWRST;`
- Then re-initialize the peripheral

### 4. Corrupted data reads
- Verify clock speed — not all devices support 400 kHz
- Check TRISE register — wrong value causes signal integrity issues
- Check pull-up resistor value — too high for the bus speed
- Verify the register address and read sequence

### 5. HAL_BUSY error return
- An ongoing transfer is still in progress
- Check for missing re-arm in interrupt callbacks
- Increase timeout values

### 6. FREQ field misconfiguration
- The FREQ field in CR2 must match the actual APB1 clock in MHz
- Wrong FREQ causes incorrect timing for glitch filter and setup/hold times

## 17. Practical Example: MPU6050 IMU

```c
#define MPU6050_ADDR     0x68
#define REG_PWR_MGMT_1   0x6B
#define REG_SMPLRT_DIV   0x19
#define REG_CONFIG       0x1A
#define REG_GYRO_CONFIG  0x1B
#define REG_ACCEL_CONFIG 0x1C
#define REG_ACCEL_XOUT_H 0x3B

typedef struct {
    int16_t ax, ay, az;
    int16_t gx, gy, gz;
    float temp;
} MPU6050_Data;

void MPU6050_Init(void)
{
    uint8_t data;
    
    // Wake up MPU6050 (clear sleep bit)
    data = 0x00;
    HAL_I2C_Mem_Write(&hi2c1, MPU6050_ADDR << 1, REG_PWR_MGMT_1,
                      I2C_MEMADD_SIZE_8BIT, &data, 1, 100);
    
    // Sample rate = 8kHz / (1 + 7) = 1kHz
    data = 7;
    HAL_I2C_Mem_Write(&hi2c1, MPU6050_ADDR << 1, REG_SMPLRT_DIV,
                      I2C_MEMADD_SIZE_8BIT, &data, 1, 100);
    
    // DLPF = 42Hz bandwidth
    data = 0x03;
    HAL_I2C_Mem_Write(&hi2c1, MPU6050_ADDR << 1, REG_CONFIG,
                      I2C_MEMADD_SIZE_8BIT, &data, 1, 100);
    
    // Gyro ±500°/s
    data = 0x08;
    HAL_I2C_Mem_Write(&hi2c1, MPU6050_ADDR << 1, REG_GYRO_CONFIG,
                      I2C_MEMADD_SIZE_8BIT, &data, 1, 100);
    
    // Accel ±2g
    data = 0x00;
    HAL_I2C_Mem_Write(&hi2c1, MPU6050_ADDR << 1, REG_ACCEL_CONFIG,
                      I2C_MEMADD_SIZE_8BIT, &data, 1, 100);
}

void MPU6050_ReadAll(MPU6050_Data *mpu)
{
    uint8_t raw[14];
    
    HAL_I2C_Mem_Read(&hi2c1, MPU6050_ADDR << 1, REG_ACCEL_XOUT_H,
                     I2C_MEMADD_SIZE_8BIT, raw, 14, 100);
    
    mpu->ax = (int16_t)((raw[0]  << 8) | raw[1]);
    mpu->ay = (int16_t)((raw[2]  << 8) | raw[3]);
    mpu->az = (int16_t)((raw[4]  << 8) | raw[5]);
    int16_t rawTemp = (int16_t)((raw[6] << 8) | raw[7]);
    mpu->gx = (int16_t)((raw[8]  << 8) | raw[9]);
    mpu->gy = (int16_t)((raw[10] << 8) | raw[11]);
    mpu->gz = (int16_t)((raw[12] << 8) | raw[13]);
    
    mpu->temp = rawTemp / 340.0f + 36.53f;
    
    // Scale: ±2g → LSB/g = 16384; ±500°/s → LSB/(°/s) = 65.5
    // float ax_g = mpu->ax / 16384.0f;
    // float gx_dps = mpu->gx / 65.5f;
}
```

## 18. Interview Questions and Answers

**Q1: Why does I2C use open-drain outputs with pull-up resistors instead of push-pull?**

**A:** Open-drain means a device can only pull the line LOW — it cannot actively drive HIGH. The pull-up resistor passively pulls the line HIGH when no device is pulling it low. This design has three key advantages: (1) **Safety** — multiple devices can drive the bus simultaneously without short-circuiting; the dominant LOW always wins (wired-AND). (2) **Arbitration** — the dominant LOW allows multiple masters to detect if another master is also driving, enabling collision-free arbitration. (3) **ACK mechanism** — the receiver simply pulls SDA LOW during the ACK clock cycle without fighting the master. With push-pull, simultaneous driving would create shoot-through current that could damage devices.

---

**Q2: How does the START and STOP condition work, and why must data not change while SCL is HIGH?**

**A:** The protocol defines two unique conditions based on SDA changing while SCL is HIGH: START is SDA falling while SCL is HIGH; STOP is SDA rising while SCL is HIGH. These conditions cannot occur accidentally during data transmission because the protocol requires SDA to remain stable while SCL is HIGH — data bits are only changed when SCL is LOW. This creates a clean separation between data bits (SDA stable during SCL high) and control signals (SDA changes during SCL high). Violating this rule during a transfer would generate a spurious START or STOP condition.

---

**Q3: What is clock stretching, and why is it important?**

**A:** Clock stretching is when a slave device holds the SCL line LOW after the master releases it. The I2C specification requires every master to check that SCL actually went HIGH before timing the next clock phase. If SCL is still low (because a slave is stretching), the master must wait. Clock stretching allows slow slaves to: (1) stall the master while preparing data to transmit, (2) process received data before being ready for the next byte. It is critical for devices like EEPROMs (which need write time) and ADCs (which need conversion time). If the master does not support clock stretching, it may read data before the slave is ready.

---

**Q4: What happens when two I2C masters try to communicate simultaneously?**

**A:** I2C has a built-in arbitration mechanism. Both masters generate START and begin transmitting their address. Each master simultaneously monitors SDA. Since open-drain signaling means LOW dominates HIGH, if master A transmits HIGH but reads LOW, it means master B is also transmitting and pulled the line LOW. Master A has lost arbitration — it immediately stops driving and waits. Master B, which transmitted LOW and reads LOW, detects no discrepancy and wins arbitration. It continues without interruption. The losing master retries after the winning master releases the bus (after its STOP condition). This is entirely hardware-handled on STM32, reported via the ARLO (Arbitration Lost) flag.

---

**Q5: Explain the I2C ACK and NACK mechanism.**

**A:** After each byte is transmitted, the transmitter releases SDA (lets it float HIGH). The receiver has one clock cycle (the 9th clock pulse) to pull SDA LOW to indicate ACK (acknowledgment — byte received successfully). If the receiver does not pull SDA LOW, SDA stays HIGH — this is NACK (not acknowledged). NACK can mean: (1) no device at that address, (2) the device is busy and cannot accept more data, (3) the receiver is the master signaling the last byte of a read sequence (master sends NACK after the last byte it wants, telling the slave to stop sending). On STM32, NACK is reported via the AF (Acknowledge Failure) flag in SR1.

---

**Q6: What is the difference between HAL_I2C_Master_Transmit and HAL_I2C_Mem_Write?**

**A:** `HAL_I2C_Master_Transmit()` sends raw bytes — a START, the address, then the provided byte array, then STOP. You control what is in the byte array entirely. `HAL_I2C_Mem_Write()` is a convenience wrapper that handles the common pattern of sending a register address followed by data. It automatically sends a START, address+write, the register address byte (1 or 2 bytes), then the data bytes, then STOP. It is equivalent to putting the register address as the first byte of the transmit buffer in `Master_Transmit`. For most sensor interactions, `Mem_Read/Write` is cleaner and avoids manual buffer construction.

---

**Q7: How do you calculate the correct pull-up resistor value for I2C?**

**A:** Two constraints must be satisfied. Lower bound: the resistor must be low enough that the driver can pull the line below V_IL_max (0.3×VCC) with its I_OL_min sink current: `R_min = (VCC - 0.4V) / I_OL`. For a typical STM32 GPIO at 3.3V with 3mA sink: R_min ≈ 967 Ω. Upper bound: the resistor × bus capacitance RC time constant must be fast enough to meet the rise time specification: Standard mode rise time < 1000ns, Fast mode < 300ns. `R_max = t_rise / (0.8473 × C_bus)`. For a 100pF bus at Fast mode: R_max = 300ns / (0.8473 × 100pF) ≈ 3543 Ω. So for 400 kHz: use 2.2–3.3 kΩ.

---

**Q8: What is a repeated START (restart) and when is it used?**

**A:** A repeated START is a new START condition issued without a preceding STOP, while the master still controls the bus. It is used in **combined transactions** — the most common example is register read: write the register address (requires WRITE direction), then read back data (requires READ direction). A STOP followed by a new START would briefly release the bus, potentially allowing another master to seize it. A repeated START maintains bus ownership through the direction change. It also signals to the slave that the transaction is continuing, which some devices require for correct operation (e.g., many sensors latch the register pointer only during a repeated-start sequence).

---

**Q9: How do you recover from a stuck I2C bus?**

**A:** If a power-cycle occurs mid-transfer, a slave may hold SDA LOW waiting for more clocks to complete its byte. Recovery: (1) issue 9 SCL pulses (via bit-banging the SCL GPIO) — this clocks out whatever partial byte the slave has, allowing it to release SDA, (2) generate a STOP condition (SDA rises while SCL is HIGH), (3) re-initialize the I2C peripheral. If SDA is stuck after 9 clocks, a hardware reset of the slave device is required. On STM32, after recovery, perform a software reset of the I2C peripheral via the SWRST bit before re-initializing.

---

**Q10: Why is there a minimum CCR value, and what happens if you set CCR too low?**

**A:** The minimum CCR value in fast mode is 1 (in DUTY=0 mode). Setting CCR too low means the SCL period becomes shorter than what the I2C hardware can reliably generate given the FREQ setting and the device's internal timing. Additionally, the electrical rise time (determined by pull-up and capacitance) may not complete within the high-phase time, causing devices to sample incorrect bit values. The TRISE register compensates for this: it tells the hardware how long the actual signal takes to rise to the logic HIGH threshold, so the hardware samples SDA correctly despite the finite rise time.
