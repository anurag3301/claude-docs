## 1. Introduction to SPI

**SPI** (Serial Peripheral Interface) is a synchronous, full-duplex, point-to-point (or one master, multiple slaves) serial communication protocol. It was developed by Motorola in the mid-1980s and is now ubiquitous in embedded systems for connecting high-speed peripherals like flash memory, ADCs, DACs, SD cards, displays, and sensors.

**Key characteristics:**
- 4-wire bus: MOSI, MISO, SCK, NSS (per slave)
- Synchronous — master provides clock (SCK)
- Full-duplex — simultaneous TX and RX
- Master-slave architecture
- No addressing — chip-select (NSS) per slave
- No built-in ACK — protocol-level error detection is device-specific
- Speeds up to hundreds of MHz (limited by device and PCB)
- Simple protocol, very low overhead

**Advantages over I2C:**
- Much faster (I2C max 400 kHz standard; SPI easily 10-80+ MHz)
- Full duplex
- Simpler timing (no START/STOP, ACK requirements)
- Lower CPU overhead at high speeds

**Disadvantages vs I2C:**
- More pins (1 CS pin per slave; I2C is 2 wires total)
- No addressing; no multi-master
- Longer wires degrade signal quality faster at high speeds

## 2. STM32F411 SPI Hardware Overview

The STM32F411 has **5 SPI peripherals**:

| Peripheral | APB Bus | Max Clock     | Features                   | Default Pins                     |
|------------|---------|---------------|----------------------------|----------------------------------|
| SPI1       | APB2    | 42 MHz        | Full-featured               | PA5(SCK), PA6(MISO), PA7(MOSI)   |
| SPI2       | APB1    | 21 MHz        | Full-featured, I2S          | PB13(SCK), PB14(MISO), PB15(MOSI)|
| SPI3       | APB1    | 21 MHz        | Full-featured, I2S          | PB3(SCK), PB4(MISO), PB5(MOSI)  |
| SPI4       | APB2    | 42 MHz        | Full-featured               | PE2(SCK), PE5(MISO), PE6(MOSI)  |
| SPI5       | APB2    | 42 MHz        | Full-featured               | PE2(SCK), PE5(MISO), PE6(MOSI)  |

SPI1 on APB2: max SPI clock = APB2_CLK / 2 = 84/2 = 42 MHz  
SPI2/3 on APB1: max SPI clock = APB1_CLK / 2 = 42/2 = 21 MHz

**Features supported:**
- 4 to 16-bit data frame size (configurable)
- MSB or LSB first
- 8 programmable baud rate prescalers (fPCLK/2 to fPCLK/256)
- 4 SPI modes (CPOL/CPHA combinations)
- Hardware or software NSS management
- TI frame format (alternative to Motorola format)
- CRC calculation and checking (hardware)
- DMA request for TX and RX
- I2S mode (audio) sharing the same peripheral

## 3. SPI Signal Lines and Electrical Characteristics

| Signal | Full Name                  | Direction (Master→Slave) | Description                       |
|--------|----------------------------|--------------------------|-----------------------------------|
| SCK    | Serial Clock               | Master → Slave           | Clock generated by master         |
| MOSI   | Master Out Slave In        | Master → Slave           | Data from master to slave         |
| MISO   | Master In Slave Out        | Slave → Master           | Data from slave to master         |
| NSS    | Negative Slave Select      | Master → Slave           | Active-LOW chip select            |

**Alternative naming (newer convention):**
- COPI = Controller Out Peripheral In (MOSI)
- CIPO = Controller In Peripheral Out (MISO)
- CS = Chip Select (NSS)

**GPIO Configuration:**
- SCK: Alternate function, push-pull, high speed
- MOSI: Alternate function, push-pull, high speed
- MISO: Alternate function, (input, no pull or pull-up)
- NSS: GPIO output, push-pull (software CS management — recommended)

**Voltage levels:** STM32F411 operates at 3.3V. For 5V SPI devices (some AVR-based boards), use level shifters.

## 4. SPI Modes — CPOL and CPHA

The SPI mode determines the clock polarity (idle state) and the edge on which data is captured.

| CPOL | CPHA | Mode | Clock Idle | Data Captured On        | Data Changed On         |
|------|------|------|------------|-------------------------|-------------------------|
| 0    | 0    | 0    | LOW        | Rising edge (1st edge)  | Falling edge (2nd edge) |
| 0    | 1    | 1    | LOW        | Falling edge (2nd edge) | Rising edge (1st edge)  |
| 1    | 0    | 2    | HIGH       | Falling edge (1st edge) | Rising edge (2nd edge)  |
| 1    | 1    | 3    | HIGH       | Rising edge (2nd edge)  | Falling edge (1st edge) |

**Definitions:**
- **CPOL (Clock Polarity):** Idle state of the clock. CPOL=0: clock idles LOW. CPOL=1: clock idles HIGH.
- **CPHA (Clock Phase):** Which edge captures data. CPHA=0: first edge captures (leading edge). CPHA=1: second edge captures (trailing edge).

**Mode 0 (CPOL=0, CPHA=0):**
```
NSS:  ‾‾‾‾\___________________________/‾‾‾
SCK:       _|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_
MOSI:     ‾‾ D7  D6  D5  D4  D3  D2 D1 D0 ‾‾
MISO:     ‾‾ D7  D6  D5  D4  D3  D2 D1 D0 ‾‾
               ↑ sampled on rising edge
```

**Which mode to use:** Check the device datasheet. Most devices specify their SPI mode. Common: W25Q flash = Mode 0 or 3; SD cards = Mode 0; many sensors = Mode 0 or 3. Modes 0 and 3 are most common because they sample on the rising edge.

## 5. SPI Data Frame Format

Unlike I2C, SPI has no address, no ACK, and no START/STOP conditions. A transfer is simply:

1. Assert NSS LOW
2. Clock out N bits on MOSI while simultaneously clocking in N bits on MISO
3. De-assert NSS HIGH

Data is shifted out MSB-first (default, configurable to LSB-first). The data width is configurable: 8-bit (most common) or 16-bit per frame.

**8-bit full-duplex exchange:**
```
NSS:   ‾\___________________________/‾
SCK:    __|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|__
MOSI:  __ D7 D6 D5 D4 D3 D2 D1 D0 _____
MISO:  __ Q7 Q6 Q5 Q4 Q3 Q2 Q1 Q0 _____
```

While the master is transmitting 8 bits (D7..D0), the slave is simultaneously transmitting 8 bits (Q7..Q0). Both devices shift simultaneously.

## 6. SPI Registers (Bare-Metal)

### SPI_CR1 — Control Register 1

| Bits  | Name    | Description                                               |
|-------|---------|-----------------------------------------------------------|
| 15    | BIDIMODE| Bidirectional mode (1=1-wire half-duplex)                 |
| 14    | BIDIOE  | Bidirectional output enable                               |
| 13    | CRCEN   | Hardware CRC calculation enable                           |
| 12    | CRCNEXT | Transmit CRC next                                         |
| 11    | DFF     | Data frame format: 0=8-bit, 1=16-bit                      |
| 10    | RXONLY  | Receive only (half-duplex)                                |
| 9     | SSM     | Software slave management (1=software NSS)                |
| 8     | SSI     | Internal slave select (when SSM=1, this is NSS value)     |
| 7     | LSBFIRST| 0=MSB first (default), 1=LSB first                        |
| 6     | SPE     | SPI enable                                                |
| 5:3   | BR      | Baud rate control (000=fPCLK/2 to 111=fPCLK/256)         |
| 2     | MSTR    | Master (1) / Slave (0)                                    |
| 1     | CPOL    | Clock polarity                                            |
| 0     | CPHA    | Clock phase                                               |

### BR (Baud Rate) Field

| BR[2:0] | Divisor | SPI Clock (APB2=84MHz) | SPI Clock (APB1=42MHz) |
|---------|---------|------------------------|------------------------|
| 000     | 2       | 42 MHz                 | 21 MHz                 |
| 001     | 4       | 21 MHz                 | 10.5 MHz               |
| 010     | 8       | 10.5 MHz               | 5.25 MHz               |
| 011     | 16      | 5.25 MHz               | 2.625 MHz              |
| 100     | 32      | 2.625 MHz              | 1.3125 MHz             |
| 101     | 64      | 1.3125 MHz             | 656.25 kHz             |
| 110     | 128     | 656.25 kHz             | 328.125 kHz            |
| 111     | 256     | 328.125 kHz            | 164.0625 kHz           |

### SPI_CR2 — Control Register 2

| Bit | Name   | Description                                          |
|-----|--------|------------------------------------------------------|
| 7   | TXEIE  | TX buffer empty interrupt enable                     |
| 6   | RXNEIE | RX buffer not empty interrupt enable                 |
| 5   | ERRIE  | Error interrupt enable                               |
| 4   | FRF    | Frame format: 0=Motorola, 1=TI                       |
| 2   | SSOE   | SS output enable (hardware NSS, master mode)         |
| 1   | TXDMAEN| TX DMA enable                                        |
| 0   | RXDMAEN| RX DMA enable                                        |

### SPI_SR — Status Register

| Bit | Name  | Description                                          |
|-----|-------|------------------------------------------------------|
| 7   | BSY   | SPI busy (transfer in progress)                      |
| 6   | OVR   | Overrun flag                                         |
| 5   | MODF  | Mode fault (NSS asserted when master)                |
| 4   | CRCERR| CRC error flag                                       |
| 3   | UDR   | Underrun flag (slave mode)                           |
| 2   | CHSIDE| Channel side (I2S mode)                              |
| 1   | TXE   | Transmit buffer empty                                |
| 0   | RXNE  | Receive buffer not empty (data ready)                |

### SPI_DR — Data Register

Reading from SPI_DR reads received data (clears RXNE).  
Writing to SPI_DR loads the TX buffer (clears TXE).

## 7. Bare-Metal SPI Configuration

```c
/*
 * Configure SPI1 in Master mode, Mode 0 (CPOL=0, CPHA=0)
 * 8-bit, MSB first, software NSS
 * SCK: PA5 (AF5), MISO: PA6 (AF5), MOSI: PA7 (AF5)
 * NSS: PA4 (GPIO output, software controlled)
 */

void SPI1_Init(void)
{
    /* 1. Enable clocks */
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;
    RCC->APB2ENR |= RCC_APB2ENR_SPI1EN;
    
    /* 2. Configure GPIO */
    // PA5 (SCK), PA6 (MISO), PA7 (MOSI) — Alternate Function 5 (SPI1)
    GPIOA->MODER &= ~((3U<<(5*2)) | (3U<<(6*2)) | (3U<<(7*2)));
    GPIOA->MODER |=  ((2U<<(5*2)) | (2U<<(6*2)) | (2U<<(7*2)));  // AF mode
    
    GPIOA->OSPEEDR |= ((3U<<(5*2)) | (3U<<(6*2)) | (3U<<(7*2)));  // High speed
    
    GPIOA->OTYPER &= ~((1U<<5) | (1U<<6) | (1U<<7));  // Push-pull
    
    // AF5 for PA5, PA6, PA7
    GPIOA->AFR[0] &= ~((0xFU<<(5*4)) | (0xFU<<(6*4)) | (0xFU<<(7*4)));
    GPIOA->AFR[0] |=  ((5U<<(5*4))  | (5U<<(6*4))  | (5U<<(7*4)));
    
    // PA4 as GPIO output for NSS (software control)
    GPIOA->MODER &= ~(3U << (4*2));
    GPIOA->MODER |=  (1U << (4*2));    // Output mode
    GPIOA->OTYPER &= ~(1U << 4);        // Push-pull
    GPIOA->OSPEEDR |= (3U << (4*2));    // High speed
    GPIOA->ODR |= (1U << 4);            // NSS initially HIGH (deasserted)
    
    /* 3. Configure SPI1 */
    SPI1->CR1 = 0;  // Clear
    
    SPI1->CR1 |= SPI_CR1_MSTR;          // Master mode
    // CPOL=0, CPHA=0 (Mode 0) — default (bits already 0)
    SPI1->CR1 |= (0x2 << SPI_CR1_BR_Pos); // BR=010: fPCLK/8 = 84MHz/8 = 10.5 MHz
    SPI1->CR1 |= SPI_CR1_SSM;           // Software NSS management
    SPI1->CR1 |= SPI_CR1_SSI;           // Internal NSS = HIGH (master won't get MODF)
    // DFF=0: 8-bit, LSBFIRST=0: MSB first (defaults)
    
    SPI1->CR2 = 0;  // No interrupts, no DMA, Motorola frame format
    
    /* 4. Enable SPI */
    SPI1->CR1 |= SPI_CR1_SPE;
}

/* Assert NSS LOW */
void SPI1_CS_Assert(void)    { GPIOA->ODR &= ~(1U << 4); }

/* De-assert NSS HIGH */
void SPI1_CS_Deassert(void)  { GPIOA->ODR |=  (1U << 4); }
```

## 8. Bare-Metal SPI Transactions

```c
/* Send and receive one byte simultaneously */
uint8_t SPI1_TransferByte(uint8_t data)
{
    /* Wait until TXE (TX buffer empty) */
    while (!(SPI1->SR & SPI_SR_TXE));
    
    /* Load data to transmit */
    SPI1->DR = data;
    
    /* Wait until RXNE (receive buffer not empty) */
    while (!(SPI1->SR & SPI_SR_RXNE));
    
    /* Read received byte */
    return (uint8_t)SPI1->DR;
}

/* Send multiple bytes, discard received data */
void SPI1_WriteBuffer(uint8_t *buf, uint16_t len)
{
    for (uint16_t i = 0; i < len; i++)
    {
        while (!(SPI1->SR & SPI_SR_TXE));
        SPI1->DR = buf[i];
        while (!(SPI1->SR & SPI_SR_RXNE));
        volatile uint8_t dummy = SPI1->DR; // Must read to clear RXNE / avoid OVR
        (void)dummy;
    }
    /* Wait until not busy */
    while (SPI1->SR & SPI_SR_BSY);
}

/* Receive multiple bytes (transmit dummy 0xFF) */
void SPI1_ReadBuffer(uint8_t *buf, uint16_t len)
{
    for (uint16_t i = 0; i < len; i++)
    {
        buf[i] = SPI1_TransferByte(0xFF);  // 0xFF = dummy TX byte
    }
    while (SPI1->SR & SPI_SR_BSY);
}

/* Full-duplex: transmit and receive simultaneously */
void SPI1_TransferBuffer(uint8_t *txBuf, uint8_t *rxBuf, uint16_t len)
{
    for (uint16_t i = 0; i < len; i++)
    {
        while (!(SPI1->SR & SPI_SR_TXE));
        SPI1->DR = txBuf[i];
        while (!(SPI1->SR & SPI_SR_RXNE));
        rxBuf[i] = (uint8_t)SPI1->DR;
    }
    while (SPI1->SR & SPI_SR_BSY);
}

/* Example: read a register from a sensor */
uint8_t SPI_ReadRegister(uint8_t regAddr)
{
    uint8_t value;
    SPI1_CS_Assert();
    SPI1_TransferByte(regAddr | 0x80);  // Set MSB for read (device-specific)
    value = SPI1_TransferByte(0xFF);    // Dummy byte to clock in data
    SPI1_CS_Deassert();
    return value;
}

/* Write a register */
void SPI_WriteRegister(uint8_t regAddr, uint8_t value)
{
    SPI1_CS_Assert();
    SPI1_TransferByte(regAddr & 0x7F);  // Clear MSB for write (device-specific)
    SPI1_TransferByte(value);
    SPI1_CS_Deassert();
}
```

## 10. HAL SPI — Polling Mode

```c
/* GPIO chip-select define */
#define CS_PORT  GPIOA
#define CS_PIN   GPIO_PIN_4

#define CS_LOW()   HAL_GPIO_WritePin(CS_PORT, CS_PIN, GPIO_PIN_RESET)
#define CS_HIGH()  HAL_GPIO_WritePin(CS_PORT, CS_PIN, GPIO_PIN_SET)

/* Transmit-only */
void SPI_Write(uint8_t *data, uint16_t len)
{
    CS_LOW();
    HAL_SPI_Transmit(&hspi1, data, len, HAL_MAX_DELAY);
    CS_HIGH();
}

/* Receive-only (sends dummy bytes) */
void SPI_Read(uint8_t *data, uint16_t len)
{
    CS_LOW();
    HAL_SPI_Receive(&hspi1, data, len, HAL_MAX_DELAY);
    CS_HIGH();
}

/* Full-duplex: transmit and receive simultaneously */
void SPI_Transfer(uint8_t *txData, uint8_t *rxData, uint16_t len)
{
    CS_LOW();
    HAL_SPI_TransmitReceive(&hspi1, txData, rxData, len, HAL_MAX_DELAY);
    CS_HIGH();
}

/* Read a sensor register */
uint8_t Sensor_ReadReg(uint8_t regAddr)
{
    uint8_t txBuf[2] = { regAddr | 0x80, 0xFF };  // Read bit + dummy
    uint8_t rxBuf[2] = { 0 };
    
    CS_LOW();
    HAL_SPI_TransmitReceive(&hspi1, txBuf, rxBuf, 2, 100);
    CS_HIGH();
    
    return rxBuf[1];  // First byte is garbage (echoed during address phase)
}

/* Write a sensor register */
void Sensor_WriteReg(uint8_t regAddr, uint8_t value)
{
    uint8_t txBuf[2] = { regAddr & 0x7F, value };  // Write bit (MSB=0)
    
    CS_LOW();
    HAL_SPI_Transmit(&hspi1, txBuf, 2, 100);
    CS_HIGH();
}
```

volatile uint8_t spiTxDone = 0;

/* DMA must be configured for SPI1_TX and SPI1_RX via CubeMX */

uint8_t txDMABuf[512];

{
    if (hspi->Instance == SPI1)
    {

### When to Use DMA for SPI

DMA is particularly valuable for SPI because SPI can run at very high clock rates. At 42 MHz, you're transferring 42M bits/second = 5.25 MB/s. Handling each byte via interrupt (8 bits = one interrupt per byte) at 5.25 MB/s means 5.25M interrupts per second — completely overwhelming the CPU. DMA handles this silently in the background.

**Use DMA for:**
- TFT/OLED display refresh (often 320×240×2 bytes = 150 KB per frame)
- Flash memory (reading large blocks)
- ADC data acquisition over SPI
- SD card operations

## 13. SPI NSS (Chip Select) Management

### Software NSS (Recommended)

The CS line is a regular GPIO output. You toggle it manually before and after each transfer. This is the most flexible approach — you can hold CS low across multiple HAL calls (multi-byte transactions with a specific protocol) and you can use any GPIO pin.

```c
// Manual CS control
#define SPI_CS_ASSERT()   HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET)
#define SPI_CS_DEASSERT() HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET)

// In CubeMX: NSS = Disabled; configure PA4 as GPIO Output
```

### Hardware NSS (Automatic)

The hardware automatically asserts NSS when SPE is set and de-asserts it after each transmission. Useful for simple single-byte transactions.

**Hardware NSS Output (master):** CR2.SSOE=1 — NSS pin goes low when SPI is enabled and transmission starts. Automatically goes high after each transmission byte.

**Hardware NSS Input (slave):** CR1.SSM=0 — the hardware reads the NSS pin. If NSS goes high mid-transfer, a Mode Fault (MODF) error is generated.

**Problem with hardware NSS:** It de-asserts between bytes — unusable for multi-byte transfers where CS must stay low throughout. **Always prefer software NSS for real applications.**

### Multi-Slave CS

```c
// Define one GPIO per slave
#define CS_FLASH_PIN   GPIO_PIN_4
#define CS_ADC_PIN     GPIO_PIN_12
#define CS_DISPLAY_PIN GPIO_PIN_15

void Flash_Deselect(void) { HAL_GPIO_WritePin(GPIOA, CS_FLASH_PIN,   GPIO_PIN_SET);   }
void ADC_Select(void)     { HAL_GPIO_WritePin(GPIOC, CS_ADC_PIN,     GPIO_PIN_RESET); }
void ADC_Deselect(void)   { HAL_GPIO_WritePin(GPIOC, CS_ADC_PIN,     GPIO_PIN_SET);   }
```

## 14. SPI Multi-Slave Wiring Patterns

### Standard (Independent CS per Slave)

```
MCU                 Slave 1
MOSI ──────────────► MOSI
MISO ◄────────────── MISO
SCK  ──────────────► SCK
CS1  ──────────────► CS (active LOW)

MCU                 Slave 2
MOSI ──────────────► MOSI
MISO ◄────────────── MISO
SCK  ──────────────► SCK
CS2  ──────────────► CS (active LOW)
```

MOSI, MISO, SCK are shared (daisy-chained or bused). Each slave has its own CS line.

### Daisy-Chain Configuration

Some devices (shift registers, some ADCs) support daisy chaining where MOSI of the master connects to MOSI of the first device, and MISO of each device connects to MOSI of the next:

```
MCU MOSI → [Slave1 MOSI→MISO] → [Slave2 MOSI→MISO] → MCU MISO
All slaves share SCK and CS
```

In daisy-chain, data is shifted through all devices simultaneously. Used for LED matrix controllers like MAX7219.

## 15. SPI as Slave

```c
/* Configure SPI1 as slave */
hspi1.Init.Mode = SPI_MODE_SLAVE;
hspi1.Init.NSS  = SPI_NSS_HARD_INPUT;  // Hardware NSS from master

HAL_SPI_Init(&hspi1);

/* Prepare data to send to master, and buffer to receive */
uint8_t slaveRxBuf[8];
uint8_t slaveTxBuf[8] = { 0xAA, 0xBB, 0xCC, 0xDD, 0xEE, 0xFF, 0x11, 0x22 };

/* Full duplex slave operation (interrupt mode) */
HAL_SPI_TransmitReceive_IT(&hspi1, slaveTxBuf, slaveRxBuf, 8);

void HAL_SPI_TxRxCpltCallback(SPI_HandleTypeDef *hspi)
{
    // Transfer complete — process slaveRxBuf, prepare next slaveTxBuf
    HAL_SPI_TransmitReceive_IT(&hspi1, slaveTxBuf, slaveRxBuf, 8);  // Re-arm
}
```

**Slave mode considerations:**
- The slave must have data ready in DR before the master starts clocking. If not, the slave will underrun.
- CS must be stable before and after the transfer.
- SPI clock is provided by the master — the slave has no control over timing.

## 16. Common Pitfalls and Debugging

### 1. Wrong SPI mode (CPOL/CPHA)
Most common mistake. Symptom: partially correct data, bit shifts. Always verify in the device datasheet which mode it requires. Use a logic analyzer with SPI decode to verify.

### 2. BSY flag polling at wrong time
The BSY flag is set when transfer is in progress AND for one SPI clock cycle after the last bit. Always wait for BSY to clear before de-asserting CS in bare-metal code:
```c
while (SPI1->SR & SPI_SR_BSY);  // Wait for complete
CS_HIGH();
```

### 3. Overrun error (OVR set)
Occurs when a new byte is received before the previous one was read. In bare-metal, always read DR after writing to it (dummy read). In HAL polling mode, always use TransmitReceive (not just Transmit) if the device sends data back.

### 4. Mode fault (MODF)
The hardware sees NSS go LOW while the peripheral is configured as master. Cause: NSS pin not properly configured (pull-up missing, or another master drives it low). Fix: use SSM=1 (software NSS) or ensure NSS pin is HIGH or not connected.

### 5. CS de-asserts between bytes (hardware NSS)
Hardware NSS de-asserts after each byte. Use software NSS and manual GPIO control for multi-byte transfers.

### 6. First byte received is garbage
Common in full-duplex: the first MISO byte corresponds to data shifted out during the command/address byte being sent. This is normal — the device responds after it knows what you want. Read the second byte (or use separate TX and RX calls).

### 7. Speed too high for device or PCB
Many modules operate at 1–4 MHz maximum. Long wires add capacitance and degrade signal. Start with a low clock (256 prescaler) and increase gradually while checking data integrity.

## 17. Practical Example: W25Q Flash Memory

```c
#define W25Q_CMD_WRITE_ENABLE  0x06
#define W25Q_CMD_READ_DATA     0x03
#define W25Q_CMD_PAGE_PROGRAM  0x02
#define W25Q_CMD_SECTOR_ERASE  0x20
#define W25Q_CMD_READ_STATUS1  0x05
#define W25Q_CMD_JEDEC_ID      0x9F

/* Read JEDEC ID */
void W25Q_ReadID(uint8_t *mfr, uint8_t *type, uint8_t *capacity)
{
    uint8_t tx[4] = { W25Q_CMD_JEDEC_ID, 0xFF, 0xFF, 0xFF };
    uint8_t rx[4] = { 0 };
    
    CS_LOW();
    HAL_SPI_TransmitReceive(&hspi1, tx, rx, 4, 100);
    CS_HIGH();
    
    *mfr      = rx[1];  // e.g., 0xEF for Winbond
    *type     = rx[2];  // e.g., 0x40 for W25Q series
    *capacity = rx[3];  // e.g., 0x17 for 128Mbit
}

/* Wait until BUSY bit clears */
void W25Q_WaitBusy(void)
{
    uint8_t tx[2] = { W25Q_CMD_READ_STATUS1, 0xFF };
    uint8_t rx[2];
    do {
        CS_LOW();
        HAL_SPI_TransmitReceive(&hspi1, tx, rx, 2, 100);
        CS_HIGH();
        HAL_Delay(1);
    } while (rx[1] & 0x01);  // BUSY bit
}

/* Read data from flash */
void W25Q_ReadData(uint32_t addr, uint8_t *buf, uint32_t len)
{
    uint8_t cmd[4] = {
        W25Q_CMD_READ_DATA,
        (addr >> 16) & 0xFF,
        (addr >> 8)  & 0xFF,
        (addr)       & 0xFF
    };
    
    CS_LOW();
    HAL_SPI_Transmit(&hspi1, cmd, 4, 100);
    HAL_SPI_Receive(&hspi1, buf, len, HAL_MAX_DELAY);
    CS_HIGH();
}

/* Write enable + page program */
void W25Q_WritePage(uint32_t addr, uint8_t *data, uint16_t len)
{
    uint8_t we = W25Q_CMD_WRITE_ENABLE;
    
    // Write Enable
    CS_LOW();
    HAL_SPI_Transmit(&hspi1, &we, 1, 100);
    CS_HIGH();
    
    // Page Program
    uint8_t cmd[4] = {
        W25Q_CMD_PAGE_PROGRAM,
        (addr >> 16) & 0xFF,
        (addr >> 8)  & 0xFF,
        (addr)       & 0xFF
    };
    
    CS_LOW();
    HAL_SPI_Transmit(&hspi1, cmd, 4, 100);
    HAL_SPI_Transmit(&hspi1, data, len, HAL_MAX_DELAY);
    CS_HIGH();
    
    W25Q_WaitBusy();
}
```

## 18. Interview Questions and Answers

**Q1: What are CPOL and CPHA? How do you determine which SPI mode to use?**

**A:** CPOL (Clock Polarity) defines the idle state of the clock: CPOL=0 means clock is LOW when idle; CPOL=1 means clock is HIGH when idle. CPHA (Clock Phase) defines which edge of the clock captures data: CPHA=0 means data is captured on the first (leading) clock edge; CPHA=1 means data is captured on the second (trailing) edge. Combining these gives four modes (0–3). To determine the correct mode, always consult the device datasheet — it will specify the SPI mode or show timing diagrams indicating clock polarity and the sampling edge. When in doubt, use a logic analyzer to observe actual timing. Mode 0 (CPOL=0, CPHA=0) is the most common.

---

**Q2: Why is there no ACK in SPI? How do you verify correct communication?**

**A:** SPI does not have an ACK mechanism because it is a point-to-point protocol with a dedicated chip-select line — there's no need to address devices, and the master fully controls the clock. Verification is device-specific: (1) some devices return a known response to a "Read ID" command (e.g., W25Q flash returns a JEDEC ID), (2) some devices have status registers you can read back, (3) you can write then read back a writable register. The master must implement application-level error detection (e.g., CRC, response validation). The STM32 SPI peripheral does support hardware CRC, which can be used for error detection on both TX and RX.

---

**Q3: What is the difference between SPI_Transmit and SPI_TransmitReceive in HAL?**

**A:** In SPI, every clock cycle simultaneously transmits and receives one bit — it is inherently full-duplex. `HAL_SPI_Transmit()` transmits data but internally still receives data, it just discards the received bytes (reads DR to clear RXNE but doesn't store it). `HAL_SPI_Receive()` wants to receive data but must transmit something to generate clocks — it transmits dummy bytes (0xFF or 0x00 depending on implementation). `HAL_SPI_TransmitReceive()` transmits a TX buffer and simultaneously stores the received bytes in an RX buffer — this is the true full-duplex operation. Use `TransmitReceive` whenever you need both the data you send and the simultaneous response from the slave.

---

**Q4: What happens if you de-assert CS before the BSY flag clears?**

**A:** The BSY flag indicates that SPI is still shifting data. In bare-metal, BSY remains set for up to one SPI clock cycle after the last bit is clocked. If you de-assert CS while BSY is set, you may cut off the last bit before the slave has fully received it, corrupting the last byte. In the worst case, the slave's internal state machine gets confused and subsequent transfers are incorrect. Always wait for BSY=0 after the last byte before de-asserting CS. Note: the HAL functions handle this internally for polling mode. In DMA mode, the `TxCpltCallback` is called after the DMA transfer is done, but BSY may still be set — add a brief delay or poll BSY before asserting CS high.

---

**Q5: What is Mode Fault (MODF) and when does it occur?**

**A:** MODF (Mode Fault) is a hardware error that occurs when: the SPI peripheral is configured as master, AND the NSS pin goes LOW (while SSM=0, hardware NSS). This indicates another master on the bus is trying to drive the SPI — a bus collision. In most practical designs with software NSS (SSM=1), MODF cannot occur because the hardware ignores the NSS pin. MODF disables the SPI (clears SPE) to prevent bus contention. To recover, read SR to clear MODF, then re-enable SPE. To prevent MODF with hardware NSS: ensure NSS is always HIGH when the peripheral is enabled as master, or switch to software NSS.

---

**Q6: How does SPI DMA work compared to SPI interrupt mode at high speeds?**

**A:** At high SPI clock rates, interrupt-driven SPI becomes impractical. At SPI1 running at 42 MHz with 8-bit frames, bytes arrive at 5.25 million bytes/second, meaning one interrupt per byte = 5.25 million interrupts per second. Each interrupt has overhead (context save/restore, handler execution) of perhaps 50–200 CPU cycles. At 84 MHz CPU, even 100 cycles/interrupt means 525 million cycles/second spent just on interrupts — exceeding the CPU's capacity. DMA solves this by having dedicated hardware transfer each byte from the SPI DR to memory without CPU involvement. The CPU receives a single interrupt only when the entire DMA transfer (potentially thousands of bytes) completes. DMA allows SPI to run at full speed while the CPU does other work.

---

**Q7: Can you run multiple SPI devices on the same bus at different speeds?**

**A:** Yes, provided you change the BRR prescaler before communicating with each device. You configure SPI for the slower device's maximum speed, communicate with it, then reconfigure for the faster device. In HAL, you can do this by modifying `hspi1.Init.BaudRatePrescaler` and calling `HAL_SPI_Init()` between transfers (which does not affect GPIO or DMA configuration). However, this adds overhead. In practice, it's simpler to run all devices at the slowest common speed, or use separate SPI peripherals for devices with very different speed requirements.

---

**Q8: What is the difference between SPI Motorola frame format and TI frame format?**

**A:** The Motorola format (default, TI mode bit = 0) is what has been described throughout — CS asserts before the first clock edge and de-asserts after the last. The TI (Texas Instruments) frame format (TI mode bit = 1) differs in that the CS pulse is generated automatically for each data frame: CS asserts for one SCK cycle to signal the start of each frame, then de-asserts, then data transfers with CS held active (or CS pulses for each byte depending on the specific TI mode variant). TI format is used by some TI peripherals. For most common SPI devices (flash, sensors, displays), use Motorola format.

---

**Q9: What is SPI bidirectional (half-duplex) mode and when is it used?**

**A:** In standard full-duplex SPI, MOSI and MISO are separate lines. In bidirectional mode (BIDIMODE=1 in CR1), only one data line is used — typically the MOSI pin serves as a bidirectional data line, and MISO is unused. The direction is controlled by the BIDIOE bit: BIDIOE=1 means output (transmit), BIDIOE=0 means input (receive). This mode reduces pin count from 4 to 3 (SCK, BIDI-data, CS). It's used by some displays (like ST7735 LCD) that communicate in half-duplex — they only receive commands/data from the MCU and never need to send back data. The tradeoff is no simultaneous TX/RX capability.

---

**Q10: How do you implement SPI with DMA for a TFT LCD display?**

**A:** TFT displays typically accept a continuous stream of pixel data. The process: (1) set the column and row address via command bytes (small transfers using polling), (2) send the write-RAM command, (3) start a DMA transfer of the entire frame buffer (e.g., 76,800 bytes for 320×240 16-bit color). Since display transfers are large and the display doesn't send back meaningful data, use `HAL_SPI_Transmit_DMA()`. In the `TxCpltCallback`, de-assert CS and optionally trigger the next frame buffer update. Use the DMA half-transfer callback to implement double-buffering — while DMA is sending the second half of the buffer, the CPU is updating the first half for the next frame. This achieves smooth animation without tearing.
